# 企业全字段 · 数据库验收清单

## 文档用途（与验证方式）

本清单用于证明：**前端企业相关页面里出现的每一个字段**（含 **详情各 Tab 展示值** 与 **各编辑弹窗/表单内的控件**）在业务上是否 **一个不漏地** 能通过 **与前端一致的请求** 写入数据库，并在 **修改后** 仍能通过 **对应编辑接口** 更新且库中可查。

**产品路径**  
列表 **「新增企业」** 仅 `POST {}` 插入空白记录并 **跳转详情**；用户可编辑的信息 **均在详情页** 通过 **`PUT` 与子资源接口** 落库。本表 **只保留「详情编辑落库」一列**，与之对应。

**主验证方式（打勾依据）**

1. **建立全量字段清单（源头）**  
   以 **前端代码** 为准（详情各 Tab、各编辑 `Modal`/`Form`），列出：控件 → API 字段 → 库列。缺映射的记为「仅前端、无落库」。详见 [企业字段-前端API库映射.md](./企业字段-前端API库映射.md)。

2. **详情编辑落库（本表唯一数据列）**  
   - 进入 **企业详情**，按 **所有 Tab** 使用 **编辑按钮/弹窗**，对凡有可编辑入口的库字段改为可辨识的新值并保存。  
   - 接口须与前端一致（`PUT /api/enterprises/{id}`、`PUT .../contacts`、`POST`/`PUT` …`/products`、`POST`/`PUT` …`/patents` 等）。  
   - **查库**（或详情接口）确认落库 → **`✅`**；详情 **无编辑入口** 或 **非本页维护** → **`—`**。  
   - 尚未实测 → **`⬜`**。

**明确不作为主路径的内容**

- **不以** 与页面脱节的随意 curl/fixtures 作为唯一依据；fixtures 仅作与前端对齐的请求体草稿。  
- **打勾以「前端字段 → 同源 API → 库核对」为准**。

**列填法**

| 符号 | 含义 |
|------|------|
| `✅` | 已按详情编辑路径完成 **库核对** |
| `⬜` | 尚未实测 |
| `—` | 详情无编辑入口、或跨系统字段非本页维护 |

**附录 B**：[企业字段-前端API库映射.md](./企业字段-前端API库映射.md)
