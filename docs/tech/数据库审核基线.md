# 数据库审核基线

> 创建日期：2026-03-22
> 用途：沉淀 TriCenter 当前“功能 -> 接口 -> 表/字段”的数据库复审入口。以后修改某个功能时，只需按命中的模块复查相关表与字段，不必每次从头梳理。

> **复审记录**：2026-03-22 已按「路由页面 → 主要 Controller/DTO → `docs/sql/tricenter_schema.sql`」完成**首轮对照**（非逐字段审计 Excel 级逐格，但覆盖主业务表与明显缺口）。此前未复审的条目已刷新；新增缺口见 §4 `DB-005`～`DB-007`。

---

## 1. 使用方式

1. 先确定本次改动的是哪个功能模块。
2. 在“当前功能到数据表映射”中找到对应模块，只复查该模块命中的表、字段、索引和关联关系。
3. 按“数据库通用复审清单”逐项核对。
4. 若表结构、字段、存储方式发生变化，直接更新本文对应模块行和“当前数据库复审重点”。

---

## 2. 数据库通用复审清单

- 页面/接口要保存的字段，数据库中是否真的存在。
- 字段类型是否匹配：`VARCHAR`、`DECIMAL`、`TINYINT`、`JSON`、`DATETIME` 是否与前后端传值方式一致。
- 字段是否允许为空，是否与表单必填规则一致。
- 新增字段是否需要默认值、历史数据兼容和导入导出兼容。
- 查询高频字段是否需要索引。
- 级联数据是否有外键或至少有明确删除策略。
- JSON 字段结构变化后，前端解析与导出逻辑是否同步更新。
- 文档变更后，是否同步更新 `docs/tech/数据库设计文档.md` 与必要的 SQL 脚本。

## 4. 当前数据库复审重点（2026-03-22）

### DB-001 合作服务的数据模型尚未闭环

- 现状：前端存在 `ServiceRecords.tsx` 和 `/enterprises/{enterpriseId}/services` 这套企业服务记录视图。
- 数据库侧：`docs/sql/tricenter_schema.sql` 中存在 `providers`、`provider_contacts`、`provider_service_areas`，但没有明确的“企业服务记录”表。
- 结论：后续做“合作服务”时，必须先确定是复用现有服务商体系，还是新增企业服务记录表，再补后端接口。

### DB-002 跨境扩展字段精简为「是否有海外分销商」— **已调整（2026-03-22）**

- 已删除列（及前后端）：`social_media_accounts`、`exhibition_history`、`overseas_distributors`（文本）、`using_crm`。
- 新增列：`has_overseas_distributors`（TINYINT）；企业详情「外贸信息」标签与编辑外贸弹窗可维护。
- 表结构以 `docs/sql/tricenter_schema.sql` 为准；已有库需自行对照 `enterprises` 表定义做增量对齐（本项目不另附该变更的独立脚本）。

### DB-003 企业主表已承载较多 JSON 字段，改动时必须重点复查序列化兼容

- 重点字段：`cross_border_platforms`、`target_markets`、`dimension_selections`、`custom_requirements` 等。
- 结论：只要修改跨境信息、需求分析、导出逻辑或 AI 报告取数逻辑，就必须同步检查 JSON 结构是否仍兼容旧数据。

### DB-004 跟进记录涉及主表状态联动

- 涉及表：`follow_up_records`、`stage_change_logs`、`enterprises`。
- 结论：只要改动跟进新增/编辑逻辑，就要同时检查是否影响漏斗阶段变更、阶段日志记录、待跟进统计。

### DB-005 企业产品表存在出口占比/利润率列，API 未暴露

- 现状：`enterprise_products` 含 `export_ratio`、`profit_margin`；`EnterpriseProduct` 实体有对应字段；`ProductCreateRequest` / `ProductUpdateRequest` **未包含**上述字段。
- 结论：当前经接口创建/更新的产品**无法写入**这两列；若业务需要，应扩展 DTO 与 Service 映射，或从表/实体中删除废弃列（需评估历史数据）。

### DB-006 专利表无「专利类型」列，与前端表单项不一致

- 现状：`enterprise_patents` 仅 `name`、`patent_no`；`PatentRequest` 亦无类型字段；企业详情专利弹窗含「专利类型」选择。
- 结论：该表单项为**纯 UI**，不入库；若需落库须新增列（及迁移）并改 DTO。

### DB-007 需求规范化相关四表未接入 Java 业务层

- 现状：`requirements`、`requirement_dimension_mapping`、`enterprise_requirements`、`enterprise_dimension_selections` 在 `tricenter_schema.sql` 中定义且可有初始化数据；`backend` 下**无**对应 `Mapper`/实体 CRUD，需求分析实际依赖 `enterprises` 上 JSON 字段 + `RequirementsConfig` 静态清单。
- 结论：属**表设计与运行路径不一致**；应择一：① 废弃/冻结四表仅作历史脚本；或 ② 逐步将读写迁到规范化表并同步前端。改动需求模块时必须显式决策，避免误以为「已走分表」。

## 5. 增量数据库复审入口

| 如果本次改动了这些功能 | 至少复查这些表/字段 |
|------------------------|--------------------|
| 企业基础信息、企业详情基本资料 | `enterprises` 的基础字段、营收字段、行业/来源/规模关联字段 |
| 联系人编辑 | `enterprise_contacts` 的主联系人标记、删除策略、回显顺序 |
| 产品/专利 | `enterprise_products`、`enterprise_patents` 的字段完整性与关联删除 |
| 跟进记录 | `follow_up_records`、`stage_change_logs`、`enterprises.stage` |
| 数据字典、行业分类、产品品类 | `system_options`、`industry_categories`、`product_categories` |
| 需求分析、企业画像 | 以 `enterprises` 的 `dimension_selections`、`removed_requirements`、`custom_requirements` 为主；若改分表策略再纳入 `requirements*` 四表 |
| 企业产品扩展指标 | `enterprise_products.export_ratio`、`enterprise_products.profit_margin` 与 `Product*Request` 是否同步 |
| 企业专利类型 | `enterprise_patents` 是否新增类型列与 `PatentRequest` |
| 合作服务 | 先确认是否新增业务表；若未定表，禁止直接实现前端落库 |
| 市场调研报告 | `market_reports` 的 `basic_report_data`、`deep_report_data` JSON 结构 |
| 海外分销商标识 | `enterprises.has_overseas_distributors` |

## 6. 下次做“数据库审核”时的建议顺序

1. 先按页面列出字段和按钮。
2. 再对照 `Controller/DTO` 看接口是否真正传这些字段。
3. 最后只核对命中的表和字段，不必全库扫描。
4. 若发现“前端有入口、接口有字段、数据库无字段”或“数据库有字段、前端无入口”，直接追加到本文“当前数据库复审重点”。

---

## 7. 维护规则

- 每次只更新本次改动命中的模块行，不需要重写全文。
- 若某模块已经完成数据库复审，可在“当前状态”中改为“已复审”，并补充日期。
- 若新增表或迁移脚本，必须同步更新 `docs/tech/数据库设计文档.md`、`docs/sql/` 和本文。
