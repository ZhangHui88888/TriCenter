# 跨系统数据连通设计文档

> 园区小程序(Booking-miniapp) × 数据分析系统(TriCenter)  
> 版本：v1.0 | 创建日期：2026-02-13

---

## 1. 背景

公司有两套独立系统：

| 系统 | 定位 | 数据库 | 技术栈 |
|------|------|--------|--------|
| **Booking-miniapp** | 园区小程序 + 后台管理 | `booking` (MySQL) | Spring Boot + Sa-Token |
| **TriCenter** | 企业数据分析系统 | `tricenter` (MySQL) | Spring Boot + JWT |

两个系统共享**服务商**和**企业用户**这两类核心实体，但各自关注不同维度的数据。系统保持独立数据库，通过关联ID字段实现数据连通。

---

## 2. 共享实体分析

### 2.1 企业用户

| | Booking | TriCenter |
|---|---|---|
| **表名** | `users` (role=1) | `enterprises` |
| **数据侧重** | C端微信用户身份（openid、头像、手机号） | 企业详细档案（资质、行业、跨境信息、漏斗阶段等50+字段） |
| **操作者** | 企业用户自己（通过小程序） | TriCenter内部管理人员 |
| **关联字段** | — | `booking_user_id` → booking.users.id |

### 2.2 服务商

| | Booking | TriCenter |
|---|---|---|
| **表名** | `providers` | `providers` |
| **数据侧重** | 运营数据（排班、预约、签到、评分、服务记录） | 档案数据（资质、服务能力、合作状态、绩效分析） |
| **操作者** | 服务商自己（小程序端）+ 后台管理员 | TriCenter内部管理人员 |
| **关联字段** | `tricenter_provider_id` → tricenter.providers.id | `booking_provider_id` → booking.providers.id |

## 4. 数据同步策略

### 4.1 同步方向

| 数据 | 方向 | 频率 | 说明 |
|------|------|------|------|
| **企业调查问卷** | Booking → TriCenter | 实时（提交即同步） | 问卷数据 → `POST /api/enterprises` 创建企业记录 |
| **数据字典选项** | TriCenter → Booking | 按需拉取 | `GET /api/options/*` → 问卷下拉选项 |
| 服务商评分 | Booking → TriCenter | 定时/事件触发 | `rating` → `service_rating` |
| 服务次数 | Booking → TriCenter | 定时/事件触发 | 统计 `provider_service_records` → `total_service_count` |
| 服务企业数 | Booking → TriCenter | 定时/事件触发 | 统计去重企业数 → `total_served_enterprises` |
| 服务商状态 | TriCenter → Booking | 手动/事件触发 | `cooperation_status=TERMINATED` 时同步禁用 Booking 端 |

### 4.2 同步方式

**阶段一**：手动关联
- 在 TriCenter 创建服务商/企业时，手动填写对应的 Booking 端 ID
- 适用于系统初期、数据量小的阶段

**阶段二（已实现 — 企业数据）**：API 接口同步
- **认证方式**：API Key（`X-API-Key` 请求头），TriCenter 通过 `ApiKeyAuthenticationFilter` 验证
- **Booking → TriCenter**：
  - `POST /api/enterprises` — 从问卷同步创建企业记录（含 `bookingUserId` 回写关联）
  - 同步状态跟踪：`enterprise_surveys.sync_status`（0=待同步, 1=已同步, 2=失败）
  - 失败重试：管理端手动触发 `/api/admin/enterprise-surveys/{id}/retry-sync`
- **TriCenter → Booking**：
  - `GET /api/options/{category}` — 拉取行业分类、规模、营收等数据字典
  - `GET /api/options/industries` — 拉取行业分类树

**阶段三（后续）**：双向 API 同步
- Booking: `GET /internal/providers/{id}/stats` — 返回服务商运营数据
- TriCenter: `GET /internal/providers/{id}/profile` — 返回服务商档案数据

### 4.3 企业调查问卷数据流

```
Booking 小程序                    Booking 后端                      TriCenter 后端
    │                                │                                  │
    │  填写4步问卷并提交              │                                  │
    ├───────────────────────────────►│                                  │
    │                                │  存入 enterprise_surveys 表       │
    │                                │  (sync_status=0)                 │
    │                                │                                  │
    │                                │  异步调用 POST /api/enterprises   │
    │                                ├─────────────────────────────────►│
    │                                │  (X-API-Key 认证)                │
    │                                │                                  │  创建企业记录
    │                                │                                  │  写入 booking_user_id
    │                                │  返回 enterprise_id              │
    │                                │◄─────────────────────────────────┤
    │                                │  更新 sync_status=1              │
    │                                │  记录 tricenter_enterprise_id    │
    │  返回提交成功                   │                                  │
    │◄───────────────────────────────┤                                  │
```

## 5. 不共享的数据

| 数据 | 所属系统 | 不共享原因 |
|------|----------|------------|
| `booking.users` (访客/管理员) | Booking | 非企业实体，TriCenter 不关心 |
| `booking.admins` | Booking | 后台审核员，与 TriCenter 用户体系无关 |
| `tricenter.users` | TriCenter | 内部管理人员，与小程序用户无关 |
| 场地/活动/时段/公告 | Booking | 纯预约业务域 |
| 行业分类/产品品类/需求/维度 | TriCenter | 纯数据分析业务域 |
| 漏斗阶段/跟进记录 | TriCenter | 内部 CRM 流程 |

---

## 6. 注意事项

1. **ID 类型一致性**：Booking 使用 `INT UNSIGNED`，TriCenter 使用 `INT`。关联字段类型应与目标表主键类型匹配
2. **不建立跨库外键**：两个数据库独立部署，关联字段仅作逻辑引用，不建立物理外键约束
3. **数据一致性**：删除操作需注意通知对方系统清理关联引用（软删除优先）
4. **安全隔离**：内部同步 API 应限制为内网访问，不暴露到公网
