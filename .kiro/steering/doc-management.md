# 文档管理规范

## 目录结构

所有项目文档统一存放在 `docs/` 目录下，按类别归入对应子目录：

```
docs/
├── README.md          # 文档导航索引（必须维护）
├── plan/              # 项目管理：里程碑、模块级任务清单
├── design/            # 需求与设计：需求文档、功能规格、交互流程（按模块拆分）
├── tech/              # 技术文档：架构、API设计、数据库、部署、运维
├── sql/               # 数据库脚本：建表与初始化数据（按序号命名）
└── test/              # 测试记录（按模块组织）
```

## 核心规则

1. **新建文档必须登记** — 每次在 `docs/` 下新建或移动文档后，必须同步更新 `docs/README.md`，在对应分类表格中添加一行，包含文件链接和简要说明
2. **归入正确子目录** — 根据文档性质放入对应子目录，禁止直接散放在 `docs/` 根目录
3. **README.md 格式** — 每个分类使用 Markdown 表格（| 文档 | 说明 |），保持链接路径正确
4. **SQL 脚本命名** — 使用数字前缀保证执行顺序（如 `01_`、`02_`），按职责拆分，不合并
5. **删除文档同步清理** — 删除任何文档后，必须从 `docs/README.md` 中移除对应条目
6. **功能变更同步更新** — 功能代码修改后，及时更新 `design/` 或 `tech/` 中相关文档
7. **AGENTS.md 体积控制** — `AGENTS.md` 总量 ≤ 2,500 Token；超出时将详细内容拆分到 `docs/` 对应子目录，`AGENTS.md` 只保留摘要和链接

## 去重规则

- **AGENTS.md 不重复 tech/** — 架构、API、数据库等详细内容只在 `docs/tech/` 中维护，`AGENTS.md` 只写摘要和链接
- **plan/ 不含技术栈** — 技术选型在 `docs/tech/技术架构文档.md`，开发计划中不重复
- **plan/ 只记模块级** — 逐接口/逐任务的进度在 `.ai-context.md`，开发计划只跟踪模块级里程碑
- **部署与运维分离** — 部署指南只写部署步骤，日常运维命令在运维手册中，不互相复制
